# Launch Readiness Checklist

This document captures everything that must be true before flipping `FEATURE_AUTH` and `FEATURE_ACCOUNT` from `hidden` to `live` on Railway. It is a living document — items should be added, removed, or reordered as launch approaches and priorities shift.

"Launch" here means making auth and accounts publicly visible on westminsterbrief.co.uk. The research tools are already live — this gate is specifically for user registration, login, paid products, and the stakeholder briefing pack.

---

## Legal and compliance

- [ ] ICO data protection registration complete; registration number added to Privacy Policy
- [ ] Privacy Policy reviewed and confirmed accurate — data controller details, third-party processors (Railway, Postmark, Google Gemini, Anthropic), ICO registration number, retention periods
- [ ] Terms of Service reviewed and includes explicit verification clause for AI-generated content (users must confirm they will verify outputs before use)

---

## Stripe and payments

- [ ] Stripe account switched to live mode; Westminster Brief branding configured (name, logo, support email)
- [ ] Products and prices created in Stripe for the stakeholder briefing pack (live mode)
- [ ] Live Stripe API keys (`sk_live_...`, `pk_live_...`) added to Railway Variables
- [ ] Stripe webhook endpoint confirmed receiving real events on production (`/stripe/webhook`)
- [ ] Checkout flow tested end-to-end with a real low-value test transaction (full purchase → webhook fires → user tier updated)

---

## Phase 2 product

- [ ] Stakeholder briefing pack build complete and tested
- [ ] At least 5 sample briefings personally generated by Mark and reviewed for quality
- [ ] At least 2 trusted external contacts (outside DfE, ideally from different audience types) have reviewed a sample briefing and given feedback
- [ ] Anthropic API key added to Railway Variables with a spend limit configured on the Anthropic dashboard

---

## Beta period

- [ ] `FEATURE_AUTH=beta` period has run for at least 1 week with selected beta users (DfE colleagues or other trusted contacts) without major issues
- [ ] Any bugs or UX issues surfaced during beta have been resolved

---

## Infrastructure and email

- [ ] `EMAIL_TEST_MODE=false` confirmed on Railway and real emails delivering successfully (welcome, password reset, deletion confirmation, data export)
- [ ] Postmark sending domain `westminsterbrief.co.uk` verified (SPF, DKIM, Return-Path) — already done locally, confirm active on Railway
- [ ] `/health` endpoint passing all checks on production (DB, Gemini API, TWFY API)
- [ ] Smoke test passing on live Railway after deploy
- [x] External Postgres backup pipeline built and first R2 upload verified — `daily/2026-04-29.sql.gz.gpg` confirmed in bucket (83MB dump, compressed to 5.5MB, encrypted)
- [ ] **[HARD BLOCKER]** Restoration test verified end-to-end before public launch. Test must include: download from R2, decrypt, decompress, restore into clean Postgres, confirm tables and row counts match production. Script and runbook are ready (`scripts/restore_from_backup.py`, `docs/recovery-runbook.md`). **No feature flag may be flipped from `hidden` to `beta` or `live` until this test passes.**

---

## Marketing and landing page

- [ ] Landing page rewritten to launch standard — honest, evidence-based copy, no overclaiming
- [ ] OG social card image deployed and rendering correctly on Twitter/LinkedIn share previews (verify with card validators, not just code inspection)

---

## Notes

Items can be parallelised — ICO registration, Stripe setup, and the beta period can run concurrently with Phase 2 build. The critical path is: Phase 2 build → sample briefing review → beta period → legal/compliance → flip flags.

Stripe and Anthropic keys should be added to Railway only once live-mode setup is complete — not before.
